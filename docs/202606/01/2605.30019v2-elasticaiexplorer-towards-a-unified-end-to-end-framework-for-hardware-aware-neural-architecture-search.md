## 摘要
神经架构搜索（NAS）已成为在任务特定和硬件特定约束下自动设计神经网络的重要方法。然而，许多现有的NAS框架将搜索空间定义、模型实现和部署流水线紧密耦合，导致难以扩展到新的硬件平台和自定义算子。本文介绍了elasticAI.explorer，一个基于Optuna的可扩展Python框架，用于硬件感知NAS。该框架引入了一种基于YAML的搜索空间规范，在采样过程中动态转换为可执行的神经网络模型。该方法支持逐层、基于单元和分层搜索空间，同时保持优化和部署的统一接口。除了架构生成，该框架还集成了硬件特定的代码生成、基于Docker的交叉编译工具链，以及设备上基准测试二进制的自动创建，从而支持硬件在环的NAS工作流程。该系统还提供了可扩展的评估器，用于FLOPs、参数数量和延迟估计。elasticAI.explorer旨在减少嵌入式AI部署的工程开销，并加速面向异构加速器平台的硬件感知NAS研究。

## Abstract
Neural Architecture Search (NAS) has become an important approach for automatically designing neural networks under task-specific and hardware-specific constraints. However, many existing NAS frameworks tightly couple search space definitions, model implementations, and deployment pipelines, making extension to new hardware platforms and custom operators difficult. In this paper, we present the elasticAI.explorer, an extensible Python framework for hardware-aware NAS built on top of Optuna. The framework introduces a YAML-based search space specification that dynamically translates into executable neural network models during sampling. The approach supports layer-wise, cell-based, and hierarchical search spaces while maintaining a unified interface for optimization and deployment. Beyond architecture generation, the framework integrates hardware-specific code generation, Docker-based cross-compilation toolchains, and automated creation of on-device benchmarking binaries, enabling hardware-in-the-loop NAS workflows. The system further provides extensible evaluators for FLOPs, parameter count, and latency estimation. The elasticAI.explorer aims to reduce the engineering overhead of embedded AI deployment and accelerate research on hardware-aware NAS for heterogeneous accelerator platforms

### 关键技术细节
1. **搜索空间DSL**：
   - 顶层结构：定义输入形状、输出形状、块序列（blocks）、全局默认参数（`default_op_params`）和复合操作（`composites`）。
   - 每个块指定候选操作（`op_candidates`）和重复模式（`type_repeat`），支持四种模式：`repeat_op`（重复相同操作，参数可变）、`repeat_params`（操作与参数仅在块内采样一次并复用）、`vary_all`（每层独立采样）、`repeat_block`（引用已有块重复）。
   - 复合操作定义可复用的子搜索空间，支持嵌套，用于实现单元化和层次化结构。
2. **动态模型实例化**：
   - 通过`ModelBuilder`根据Optuna采样参数动态构建PyTorch模型，自动插入适配模块（adapter）以处理不同层间的形状不兼容，无需手动编写转换逻辑。
   - 通过`@register_layer("op_name")`装饰器注册新操作，实现可扩展性；适配器通过转换注册表管理。
3. **评估API**：
   - 分为性能估计器（如准确率）和成本估计器（FLOPs、参数量、延迟）。
   - 支持多目标优化（直接使用Optuna多目标算法）或通过`OptimizationCriteria`将目标、软约束、硬约束组合，采用加权和或自定义聚合函数。
   - 硬约束优先检查，无效架构提前终止，节省计算资源。
4. **硬件部署流水线**：
   - 生成器（Generator）将模型转化为目标特定格式（如TorchScript用于Raspberry Pi，LiteRt用于Pico，elasticAI.creator用于FPGA）。
   - 支持Docker交叉编译工具链，确保可复现性。
   - 两种模式：①对最终最优模型部署并测量；②在NAS过程中启用硬件在环，将实时测量结果（延迟等）反馈给优化循环。
   - 生成器可通过反射API通知模型生成器关于硬件支持的操作和实现，避免采样不支持的操作。

### 算法流程（文字说明）
1. 用户编写YAML搜索空间配置文件（可选包含预处理配置）。
2. 搜索空间翻译器将YAML转换为Optuna兼容的搜索空间。
3. Optuna采样器建议超参数（包括块操作、重复深度、层参数等）。
4. `ModelBuilder`根据采样参数动态构建并实例化模型（若含预处理，则同时构建预处理管线）。

6. Optuna更新采样器，重复步骤3-5直到达到终止条件。
7. 选择最优模型，通过生成器部署到目标硬件，完成端到端流程。
